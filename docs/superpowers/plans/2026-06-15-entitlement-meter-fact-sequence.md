# Entitlement Meter Fact Sequence Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a server-owned ordering field to priced entitlement meter facts so Tinybird analytics can pick the Durable Object's final applied balance without relying on customer-provided idempotency keys.

**Architecture:** Keep the current Tinybird `idempotency_key` fallback in place for existing rows. Add an additive `fact_sequence` field to the meter fact contract, stamp it in `EntitlementWindowDO` when facts are produced, persist the counter in the DO's SQLite meter state, and update only the analytics rank expression to prefer `fact_sequence` while preserving the existing AggregateFunction state type.

**Tech Stack:** TypeScript, Zod, Cloudflare Durable Objects, Drizzle SQLite migrations, Tinybird pipes/datasources/tests, pnpm, Vitest.

---

## First Principles

Today the analytics rollup asks Tinybird for the latest cumulative `amount_after`. The current fix makes that deterministic with:

```sql
tuple(timestamp, created_at, idempotency_key)
```

That is invoice-consistent today because the ingestion processor also sorts equal event timestamps by idempotency key. But the idempotency key is customer-provided, so it should not be the long-term analytics ordering invariant.

This plan adds:

```txt
fact_sequence = server-owned counter assigned by EntitlementWindowDO
```

Then Tinybird ranks rows with:

```sql
tuple(timestamp, created_at, sequence_rank)
```

where `sequence_rank` encodes `fact_sequence` first and falls back to `idempotency_key` for old rows. This keeps the current fix working while new facts stop depending on a customer-controlled field.

Important scope boundary: this plan does **not** change the DO's event application order. It only records the order the DO actually applied/priced facts. Changing event application order for equal timestamps is a separate billing behavior decision and should be planned independently.

## Tradeoffs

- Pros: Tinybird no longer needs to trust a customer-provided idempotency key to identify the final applied balance for new facts.
- Pros: The current idempotency-key fallback keeps old rows and current production analytics safe during rollout.
- Pros: The Tinybird AggregateFunction state type stays `Tuple(Int64, Int64, String)`, so this avoids a destructive aggregate-state migration.
- Cost: One small DO SQLite schema migration is required.
- Cost: This does not change same-timestamp rating order. It records existing DO application order; it does not redefine billing semantics.

```ts
private applyAndPriceSingleApplyEvent(
  tx: DrizzleSqliteDODatabase<typeof schema>,
  params: {
    activeGrants: ActiveGrantInput[]
    createdAt: number
    entitlement: EntitlementConfigInput
    input: ApplyInput
    meter: MeterIdentity
    metrics: SingleApplyExecutionMetrics
    overageStrategy: OverageStrategy
  }
): { meterState: MeterStateDraft; pricedFacts: PricedFact[] } {
  const { activeGrants, createdAt, entitlement, input, meter, metrics, overageStrategy } = params
  const { facts, meterState } = this.applySingleMeterEventInTransaction({
    activeGrants,
    createdAt,
    entitlement,
    input,
    meter,
    metrics,
    overageStrategy,
    tx,
  })

  const priced = this.priceFactsFromCompactGrantState(tx, {
    activeGrants,
    entitlement,
    eventTimestamp: input.event.timestamp,
    facts,
  })
  metrics.pricedFactCount = priced.pricedFacts.length
  metrics.grantAllocationCount = priced.touchedStateCount
  metrics.grantWindowWriteCount = priced.periodWriteCount

  return { meterState, pricedFacts: priced.pricedFacts }
}
```
